# B+ 트리 인덱스 구현 핵심 구성

## 1. 자동 ID 기반 인덱스 구조

레코드가 추가될 때마다 ID가 자동으로 부여되어야 한다. 이 ID는 B+ 트리의 key가 되고, value에는 해당 레코드를 다시 찾을 수 있는 위치 정보를 저장해야 한다.

메모리 기반 DB라면 value는 `Record *` 또는 레코드 배열의 `row_index`처럼 단순한 참조로 둘 수 있다. 중요한 점은 `id`를 알면 전체 테이블을 순회하지 않고 바로 레코드 위치를 얻을 수 있어야 한다는 것이다.

## 2. B+ 트리 삽입과 split 처리

B+ 트리는 대량의 INSERT 이후에도 검색 성능을 유지하기 위해 균형 잡힌 트리 구조를 유지해야 한다. 이를 위해 삽입 시 leaf node가 가득 차면 split하고, 분리 기준 key를 parent node에 반영해야 한다.

parent node도 가득 차면 internal node split이 발생하며, root까지 split되면 새로운 root를 만들어야 한다. 이 과정이 올바르게 동작해야 1,000,000개 이상의 레코드를 삽입해도 `WHERE id = ?` 검색을 빠르게 수행할 수 있다.

## 3. SQL 처리기와 인덱스 실행 경로 연결

기존 SQL 처리기는 그대로 사용하되, `WHERE id = ?` 조건일 때만 B+ 트리 인덱스를 사용하도록 실행 경로를 분기해야 한다.

ID 기준 SELECT는 B+ 트리에서 record 위치를 찾고, 다른 필드 기준 SELECT는 기존처럼 선형 탐색을 수행한다. 이 분기가 있어야 인덱스 검색과 선형 탐색의 성능 차이를 비교할 수 있다.

## 전체 흐름

```text
INSERT 실행
-> 자동 ID 부여
-> 테이블에 레코드 추가
-> B+ 트리에 id와 record 위치 등록

SELECT 실행
-> WHERE 조건 확인
-> id 조건이면 B+ 트리 검색
-> 다른 필드 조건이면 선형 탐색
```

이 3가지가 B+ 트리 인덱스 구현을 기존 C 기반 미니 DB에 연결하기 위한 핵심 단위다.
