# B+ 트리 인덱스 구현 핵심 구성

## 1. ID 컬럼과 파일 offset 기반 인덱스 엔트리

`WHERE id = ?` 검색에서 B+ 트리 인덱스를 사용할 수 있어야 한다. 인덱스는 레코드의 `id` 값과 해당 레코드를 다시 찾기 위한 파일 내 위치 정보를 연결한다.

레코드가 추가되면 그 레코드의 `id`가 인덱스에 등록되어야 한다. 이후 같은 `id`로 검색할 때는 전체 파일을 처음부터 끝까지 확인하지 않고, 인덱스가 가리키는 위치의 레코드를 찾을 수 있어야 한다.

## 2. B+ 트리 삽입과 split 처리

B+ 트리는 대량의 INSERT 이후에도 검색 성능을 유지할 수 있어야 한다. 레코드가 계속 추가되어도 트리의 높이가 한쪽으로 치우치지 않고 균형 잡힌 상태로 유지되어야 한다.

이 구조가 유지되어야 1,000,000개 이상의 레코드를 삽입한 뒤에도 `WHERE id = ?` 검색을 빠르게 수행할 수 있다. 이 문서에서는 세부 구현 방식보다, 대량 데이터에서도 인덱스가 안정적으로 검색 성능을 제공하는 것을 핵심 동작으로 본다.

## 3. SQL 실행 경로 분기와 성능 비교

기존 SQL 처리기는 그대로 사용하되, `WHERE id = ?` 조건일 때만 B+ 트리 인덱스를 사용하도록 실행 경로를 분기해야 한다.

ID 기준 SELECT는 인덱스를 통해 레코드를 찾고, 다른 필드 기준 SELECT는 기존처럼 파일 전체를 선형 탐색한다. 이 두 경로가 분리되어 있어야 인덱스 검색과 선형 탐색의 성능 차이를 비교할 수 있다.

## 전체 흐름

```text
INSERT 실행
-> INSERT 값에서 id 추출
-> 테이블 파일에 레코드 추가
-> B+ 트리에 id와 레코드 위치 등록

SELECT 실행
-> WHERE 조건 확인
-> id 조건이면 B+ 트리 인덱스로 레코드 검색
-> 다른 필드 조건이면 선형 탐색
-> 두 검색 방식의 실행 시간 비교
```

이 3가지가 B+ 트리 인덱스 구현을 기존 C 기반 미니 DB에 연결하기 위한 핵심 단위다.
