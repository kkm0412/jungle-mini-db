# B+ 트리 인덱스 구현 핵심 개념

## 1. 인덱스

인덱스는 특정 컬럼 값을 기준으로 레코드를 빠르게 찾기 위한 보조 구조다. 이 과제에서는 `id` 값을 기준으로 레코드 위치를 찾는 것이 핵심이다.

인덱스가 없으면 `WHERE id = ?` 검색도 파일의 모든 레코드를 순서대로 확인해야 한다. 인덱스가 있으면 `id`를 통해 필요한 레코드 위치를 먼저 찾고, 해당 레코드만 읽을 수 있다.

### 1.1 인덱스 key

인덱스 key는 레코드를 찾기 위해 기준으로 삼는 값이다. 이 과제에서는 `id` 컬럼 값이 key가 된다.

### 1.2 인덱스 value

인덱스 value는 key로 찾은 결과가 실제 레코드와 연결되도록 돕는 값이다. 파일 기반 DB에서는 레코드 자체보다 파일 안의 레코드 위치가 value 역할을 한다.

### 1.3 인덱스 검색

인덱스 검색은 key를 이용해 필요한 레코드 위치를 찾는 과정이다. 이 검색이 성공하면 전체 파일을 순서대로 훑지 않고도 원하는 레코드에 접근할 수 있다.

## 2. B+ 트리

B+ 트리는 많은 key를 정렬된 상태로 관리하면서도 검색 경로를 짧게 유지하는 트리 구조다. 대량의 데이터가 들어와도 한쪽으로 치우치지 않도록 균형을 유지한다.

이 과제에서 B+ 트리는 `id` 값을 빠르게 찾기 위한 인덱스 구조로 사용된다. 많은 레코드가 추가된 뒤에도 `WHERE id = ?` 검색이 전체 탐색보다 빠르게 동작해야 한다.

### 2.1 정렬된 key 관리

B+ 트리는 key를 정렬된 상태로 유지한다. 정렬되어 있어야 검색할 때 어느 방향으로 이동해야 하는지 빠르게 판단할 수 있다.

### 2.2 균형 유지

B+ 트리는 데이터가 많아져도 특정 방향으로만 길어지지 않아야 한다. 균형이 유지되어야 많은 레코드가 있어도 검색 경로가 짧게 유지된다.

### 2.3 leaf 중심 검색 결과

B+ 트리에서는 실제 검색 결과가 leaf 영역에서 결정된다. `id` 검색도 최종적으로 leaf에서 해당 key와 연결된 레코드 위치를 찾는 흐름이 된다.

## 3. 파일 내 레코드 위치

파일 기반 DB에서는 레코드가 메모리 안에만 존재하지 않고 파일에 저장된다. 따라서 인덱스는 레코드 자체가 아니라, 파일 안에서 해당 레코드를 다시 찾을 수 있는 위치 정보를 가지고 있어야 한다.

`id` 검색 결과로 파일 내 위치를 얻을 수 있어야 인덱스 검색이 실제 SELECT 결과로 이어진다. 즉, B+ 트리 검색 결과는 최종 데이터가 아니라 파일에서 레코드를 읽기 위한 단서다.

### 3.1 파일 저장 순서

레코드는 INSERT되는 순서대로 파일에 저장된다. 같은 파일 안에서 각 레코드는 서로 다른 위치를 가진다.

### 3.2 레코드 위치 정보

레코드 위치 정보는 파일 안에서 특정 레코드를 다시 찾기 위한 기준이다. 인덱스는 `id`와 이 위치 정보를 연결해야 한다.

### 3.3 위치 기반 읽기

위치 기반 읽기는 파일 전체를 탐색하지 않고 필요한 레코드가 있는 지점으로 이동해 데이터를 읽는 개념이다. 이 동작이 가능해야 인덱스 검색 결과가 실제 SELECT 결과로 이어진다.

## 전체 관계

```text
id 값
-> B+ 트리 인덱스 검색
-> 파일 내 레코드 위치 확인
-> 해당 위치의 레코드 읽기
```

이 3가지 개념이 `WHERE id = ?` 검색을 선형 탐색이 아닌 인덱스 검색으로 바꾸기 위해 필요한 최소 개념이다.
